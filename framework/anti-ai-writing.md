# Anti-AI Writing パターン辞書

> 関連: [[skill-contract]] / [[review-policy]] / [[templates/README|templates]]

3gpp-research の発表資料・研究ノートで **避けたい日本語表現** の辞書。
LLM が生成した文章で頻発するパターンを列挙し、置換指針を示す。

主な参照スキル:
- `/build-presentation` — 話者原稿生成と Phase 4 自己レビューで本辞書を参照
- 任意の `documents/` ノート編集時 — 自己レビューでスクリーニング

> 出典: claude-scholar の `writing-anti-ai` を日本語向けに翻案。
> Wikipedia "Signs of AI writing" を原典とする。

## なぜこの辞書が必要か

LLM は「最も統計的にありそうな表現」を生成する。
結果として、特定のパターン（比喩・形容詞インフレ・vague attribution・rule of three）が高頻度で出現し、
**情報量はほぼゼロなのに見栄えのする文** が量産される。

3gpp-research の方針（[[skill-contract]] Global Will Not §1-§4 と [[feedback_presentation_quality]]）と直接衝突するため、
スキル実行中に意識的に検出・除去する。

## §1. 禁止パターン辞書

### §1.1 比喩で逃げる表現

技術的内容を直接記述せず、比喩で意味を曖昧化する。

| 禁止 | 例 | 置換 |
|:---|:---|:---|
| 〜の証として / 〜の象徴として | 「Rel-15 は 5G の幕開けの象徴である」 | 「Rel-15 で NR の物理層仕様が確定した」 |
| 重要な役割を果たす | 「DCI は制御の重要な役割を果たす」 | 「DCI はスケジューリング情報を運ぶ」 |
| 橋渡しする / 鍵を握る / 扉を開く / 柱となる | 「AI/ML が 6G の鍵を握る」 | 「AI/ML を物理層に組み込む議論が Rel-19 で始まった」 |
| 〜を支える / 〜の心臓部 | 「OFDM が NR の心臓部」 | 「NR の DL/UL 波形は OFDM ベースで定義されている」 |
| ナビゲートする / 駆動する（人間以外を主語） | 「アルゴリズムが探索をナビゲートする」 | 「アルゴリズムが探索する」 |
| 〜の世界 / 〜の風景 | 「6G の世界では…」 | 削除 |

### §1.2 vague attribution（出典なき権威付け）

主語が不明な「専門家」「業界」を使って事実を装う。

| 禁止 | 置換 |
|:---|:---|
| 専門家は〜と指摘する / 専門家によれば | 具体的な出典・著者・寄書番号を書く。書けないなら主張を取り下げる |
| 業界では〜と言われている | 「TSG-RAN1 #124 の議事録では〜と記録されている」のように一次情報を引く |
| 一般的に〜とされている | 「[[references/TS38.211-v18.6.0]] §4.3.2 では〜と定義されている」 |
| 多くの研究者が〜を指摘 | 「<著者名> et al. (年) は〜と報告した」 |
| 広く知られているように | 削除。事実なら根拠を、不要なら省略する |

### §1.3 rule of three の機械的適用

3項目を強引に揃える癖。実際には2項目で十分なことも多い。

- 「速度・容量・遅延の3要素」のように **意味が薄いまま3つ並べる** のは禁止
- 4要素以上ある場合に「3つに絞る」必要はない。出典が4要素ならそのまま4つ書く
- 2要素で説明できるなら2要素で書く

**判定基準**: 各項目が独立に MECE か、根拠は一次情報のどこにあるか。曖昧なら削除。

### §1.4 em-dash と「〜だけでなく〜である」の連発

| 禁止 | 例 | 置換 |
|:---|:---|:---|
| em-dash の連発 | 「DCI は — 短いが情報密度の高い — 制御チャネルである」 | 「DCI は短く情報密度が高い制御チャネルである」 |
| 〜だけでなく〜でもある | 「Rel-15 だけでなく Rel-16 でも採用された」 | 「Rel-15 と Rel-16 の両方で採用された」 |
| It's not just X, it's Y 的構文 | 「これは単なる規格変更ではなく、思想転換である」 | 「規格変更の背景に〜という設計思想の変化がある」 |
| ネガティブ並列（「〜ではなく〜である」の連打） | 同上 | 1回に留める |

### §1.5 形容詞インフレ

定量根拠なしの強調。

| 禁止 | 置換 |
|:---|:---|
| 画期的な / 革新的な / 不可欠な / 極めて重要な | 削除。または「Rel-15 で初めて導入された」のような事実に置換 |
| 飛躍的に / 劇的に / 大幅に | 「2倍に」「30% 改善」のように数値で書く |
| 非常に / 極めて / かなり | 削除。または数値+単位+基準 |
| 多くの / 少ない / 一部の | 「N 件中 M 件」のように具体化 |
| 高い / 低い / 大きい / 小さい | 数値+単位+比較対象 |
| 約 / およそ（範囲なし） | 具体値または明示的な範囲（例: 約 5ms → 4-6ms） |

### §1.6 演出的接続・前置き

情報量ゼロの接続。

| 禁止 | 置換 |
|:---|:---|
| ここで重要なのは / 注目すべきは | 削除。事実をそのまま書く |
| 興味深いことに / 驚くべきことに | 削除。結果を提示する |
| 言うまでもなく / 当然のことながら | 削除 |
| まず理解すべきは | 削除。説明を始める |
| 本スライドでは〜を説明する | 削除。スライドタイトルが内容を示す |

### §1.7 根拠のない評価・常套句

| 禁止 | 置換 |
|:---|:---|
| 効果的 / 有用 / 有望 | 何がどう変わるかを具体的に書く |
| 近年〜が注目されている（出典なし） | 「<年> 以降、TSG-RAN1 では〜が継続議題」のように年と一次情報を示す |
| 〜が期待される（無主語） | 主語と根拠を書く。書けないなら削除 |

### §1.8 繰り返し・冗長

- 同じ主張を言い換えて複数回書かない
- 「前述のとおり〜」で再説明しない（セクション番号で参照）
- 「新しく新規な」のような同義反復は片方を削除

### §1.9 造語・未定義用語

- 引用根拠のない専門用語風の造語を使わない
- 略語は **初出時に正式名称と定義** を提示する
  - 例: 初出 「DCI (Downlink Control Information)」、以降は DCI

## §2. 命名根拠を必ず示す

[[feedback_presentation_quality]] で確認済みの方針:

- 「最小」「共通」「拡張」等の修飾語を使うときは **なぜそう呼ぶか** を併記する
  - 例: 「**最小** SCS = 15 kHz（Rel-15 で最初に定義された SCS だから "最小"）」
- 仕様用語と一般用語が混在する場合は **どちらを使ったか** を明示
  - 例: 「ここでの "サブキャリア間隔" は TS 38.211 の SCS（Subcarrier Spacing）を指す」
- 数式記号や変数名も初出時に意味を書く

## §3. 自己チェック手順（生成後に実行）

文章を書き終えたら **必ず以下の grep ベース検査を行う**:

```bash
# §1.1 比喩で逃げる表現
grep -nE "(の証|の象徴|重要な役割|鍵を握る|橋渡し|心臓部|世界|風景|ナビゲート|駆動)" <ファイル>

# §1.2 vague attribution
grep -nE "(専門家|業界では|一般的に|多くの研究者|広く知られて)" <ファイル>

# §1.4 連発系
grep -nE "(だけでなく|単なる.*ではなく)" <ファイル>

# §1.5 形容詞インフレ
grep -nE "(画期的|革新的|不可欠|極めて重要|飛躍的|劇的|大幅に|非常に|極めて|かなり)" <ファイル>

# §1.6 演出的接続
grep -nE "(ここで重要|注目すべき|興味深い|驚くべき|言うまでもなく|当然のこと|まず理解|本スライドでは.*を説明)" <ファイル>

# §1.7 常套句
grep -nE "(効果的|有用|有望|近年.*注目|期待される)" <ファイル>
```

**運用ルール**:

- 検出件数を「ベースライン」として記録する（最初は 0 件を狙わない）
- 各検出箇所について「残す／削除／置換」を明示的に判断する
- 機械的に grep で 0 件にすることが目的ではない。**意味のある日本語文** を残す
- 文体が硬くなりすぎたら一部の表現は意図的に残す（演出的でなく説明的なものは可）

## §4. 「魂のない文章」の回避

パターンを除去するだけでは「無声・中立・退屈」な文になる。
研究ノートの場合、これも避ける（ただし発表資料側の話者原稿に限定）。

| 兆候 | 対処 |
|:---|:---|
| 全文が同じ長さ・構造 | 短い文と長い文を混ぜる |
| 意見ゼロ、淡々と事実だけ | 「ここは仕様の解釈に幅がある」「私はこの主張に懐疑的」のような **判断** を入れる |
| 第一人称ゼロ | 自分の解釈は「私は〜と読んだ」「ここは未検証」と一人称で書く |
| 不確実性の表明ゼロ | 「[要確認]」「confidence: low」を恐れず使う |

ただし `documents/` の本文はあくまで研究記録なので、過剰な感情表現は避ける。
「魂を入れる」のは話者原稿・ブログ的記述のみ。
