## 1. Waveform（6GR）

### 1.1 アジェンダ位置と背景

- **AI 番号**: 10.2.1（Waveform）, 10.2.x（PAPR・Modulation 系のサブアジェンダ）
- 5G NR ベースライン継承が #122 で大枠合意済（CP-OFDM DL / DFT-s-OFDM UL、SCS は 15/30/120 kHz 等）（[6Ghow](https://www.6ghow.com/first_6G_agreements/), [MCNS](https://mcns5g.com/first-6g-agreement-reached-cp-ofdm-and-dft-s-ofdm-confirmed-but-is-this-really-the-future-of-6g/)）。
- 「**新規 waveform は不排除**（new waveforms are not precluded）」— 現状は ZAC OTFS / AFDM 等が Jio らから提案、シミュレーション不足を理由に保留中。

### 1.2 #124 → #124bis の進展

| 項目 | #124（Gothenburg, 2026-02） | #124bis（Malta, 2026-04） |
|:---|:---|:---|
| **PAPR 削減候補** | リスト化合意：FDSS（baseline / spectrum extension / spectrum truncation for π/2 BPSK）、tone reservation、GMSK-approximation FDSS、π/2-PAM・offset-QAM 系 modulation 派生、AI/ML ベース | 評価フェーズ。NR Rel-15 DFT-s-OFDM をベースラインに性能比較データが集まり始める |
| **Multi-rank UL MIMO for DFT-s-OFDM** | high-priority study に指定 | rank-2 以上の達成可能性を評価中 |
| **企業対立軸** | Samsung・vivo（AI/ML PAPR 強化派）vs Qualcomm（従来信号処理派）。両アプローチを並列評価する妥協 | 引き続き並列評価 |

### 1.3 何が問題視されているか（Pros / Cons）

| 観点 | 利点（Pro） | 欠点（Con） |
|:---|:---|:---|
| **OFDM 継承** | 5G エコシステム再利用、MRSS（5G-6G スペクトル共有）が容量損失 <5% で実現可 | 「6G らしさ」が出ない、革新性への懐疑が業界からも提起 |
| **AI/ML PAPR 削減** | 従来 FDSS 比でカバレッジ向上の可能性 | チップ実装複雑度、UE 電力、モデル汎化性が未確認 |
| **新規波形（OTFS 等）** | ドップラー耐性・ NTN 親和 | シミュレーション量不足、エコシステム形成負荷 |

### 1.4 #125 の見どころ

- PAPR 削減候補の **down-selection** が進む可能性
- Multi-rank UL MIMO の **実装可能 rank** に対する各社の評価結果

> **詳細別ノート**: [[260501_6GR-Waveform-PAPR削減_変遷]]（#124bis で 11 候補 → 3 候補に down-select 完了、Cohere の Zak-OTFS は noted、企業マトリクス + SEP 仮説）

## 2. Channel coding（6GR）

### 2.1 アジェンダ位置と背景

- **AI 番号**: 10.3.1（6G channel coding）。FL: AccelerComm / ETRI（draft 文書名から推定、要確認）。
- 5G の LDPC（data）+ Polar（control）が大枠ベースライン。**新規符号は性能向上を実証した上で初めて検討**という慎重路線。

### 2.2 #124 → #124bis の進展

| 観点 | 概要 |
|:---|:---|
| **論点** | 短ブロック性能、複雑度・遅延、ハードウェア再利用性、ML 復号 |
| **#124 主要 Tdoc** | [[R1-2600522]]（EURECOM）— "Discussion on Channel Coding for Small Block Lengths"。AI 10.3.1。短ブロックでの LDPC/Polar の劣化と 6G の URLLC/MTC 用途への対応を議論 |
| **状況** | 「**core parts almost confirmed**（核は概ね確定）」と Apex Standards が要約。新候補（GLDPC-PC 等）は学術段階、3GPP 採用への道筋は未明 |

### 2.3 何が問題視されているか

- **短ブロック性能**: 5G LDPC は中〜長ブロック最適、極短は Polar/RM 系が強い。6G URLLC・A-IoT 級では境界が曖昧
- **AI/ML 復号**: 性能向上は文献多数だが、**実装複雑度 / 推論遅延 / モデル更新運用** が標準化の壁
- **CRAN 親和性**: クラウド RAN 実装で復号負荷が CN 集約に乗る場合、計算リソース割当が競争要因（[arXiv 2508.08225](https://arxiv.org/html/2508.08225v1)）

### 2.4 #125 の見どころ

- 短ブロック向け新方式の **評価メトリクス合意**（BLER + 偽警報率の閾値）
- LDPC / Polar の拡張範囲（lifting size、kernel）の絞り込み

## 3. Multi-antenna system / MIMO（6GR + Rel-20 NR MIMO Phase 6）

> ユーザー指定の「Multi-antenna system」と「MIMO」は同一アジェンダ群を指す。Rel-19 までの **NR MIMO** と、6GR の **新規 MIMO 設計**（10.5 系）が並走。

### 3.1 アジェンダ位置と背景

- **6GR 10.5 系（Multi-antenna umbrella）**: IA / 同期 / PRACH / BWP / **MIMO operation（10.5.2）** / **CSI** / **Control & HARQ** を含む統合フレームワーク。
- **10.5.2 サブ AI**:
  - 10.5.2.1 DL 制御チャネル（PDCCH 物理層）
  - 10.5.2.2 DL 共有チャネル（PDSCH）
  - 10.5.2.3 UL チャネル（PUSCH/PUCCH）
  - 10.5.2.4 Beam Management
- **Rel-20 NR MIMO Phase 6**: 5G-Adv の枠組みでの拡張継続。

### 3.2 #124 → #124bis の進展

| 観点 | 内容 |
|:---|:---|
| **アンテナ数の桁** | Rel-15 の 1024〜4096 element クラスから、6GR は **256〜512+ CSI port** へスケール提案（[arXiv 2508.08225](https://arxiv.org/html/2508.08225v1) Table III） |
| **D-MIMO（分散 MIMO）** | 「**業界の採用は遅い・消極的**」とされる。バックホール分散・位相同期の実装難 |
| **コードブック改善** | 5G Type-II 強化で約 **20% spectral efficiency 増** が現実的な狙い |
| **UL MIMO 強化** | DFT-s-OFDM での coherent multi-layer 伝送が新たな焦点 |
| **Duplexing** | Full-duplex FDD、sub-band full-duplex（gNB / UE）も #122 から候補入り |

### 3.3 何が問題視されているか

| 観点 | Pro | Con |
|:---|:---|:---|
| **5G フレームワーク継承** | 既存 codebook / DMRS 設計再利用、ベンダー実装コスト低 | 6G らしい飛躍が出にくい |
| **D-MIMO** | 理論的にはセル境界性能・干渉抑制で大幅優位 | バックホール bandwidth、位相同期、O&M 複雑度 |
| **大規模アンテナ** | スループット拡大、ビーム解像度向上 | CSI 取得帯域の急増、UE 計算負荷、SRS 設計の見直しが必要 |

### 3.4 #125 の見どころ

- 10.5.2.1（PDCCH）の伝送方式の絞り込み（特に共有 search space の構造）
- 10.5.2.3（UL）の DMRS / SRS の初期構造合意

> **詳細別ノート**: [[260428_RAN1-124bis-MIMO-operation-10.5.2]]（10.5.2.1〜10.5.2.3 の 124bis 詳細）

## 4. Initial Access（IA, 6GR）

### 4.1 アジェンダ位置と背景

- **6GR 10.5 系の IA サブアジェンダ**（IA / 同期 / PRACH）として包括。AI 番号は要確認。
- 5G NR の SS/PBCH (SSB) + 4-step / 2-step RACH の継承が出発点。

### 4.2 #124 → #124bis の進展

| 観点 | 内容 |
|:---|:---|
| **論点** | (a) SSB の SCS / 配置、(b) PRACH preamble 構造、(c) NTN 親和（large delay / Doppler）、(d) ML 補助のセル探索 |
| **開示状況** | Apex Standards が「IA / 同期 / PRACH」がアジェンダに含まれることを示すのみ。**具体合意の二次情報は未確認** |
| **MRSS 文脈** | 5G-6G コロケート/非コロケート展開での IA 共通化が RP#111 から RAN1 タスク化、評価 deadline は 2026-09（RAN#113） |

### 4.3 何が問題視されているか

- **NTN 親和**: 衛星リンクの伝搬遅延・周波数オフセットを既存 RACH preamble が吸収しきれるか
- **MRSS（5G-6G スペクトル共有）**: SSB 周期と LTE-NR DSS で得た教訓の取り込み（"lessons learned from LTE-NR DSS" を学ぶ合意あり）
- **エネルギー効率**: idle UE のセル探索電力が大規模アンテナ時代に再評価される

### 4.4 #125 の見どころ

- SSB / PRACH 構造の **初期方向性**（具体的 numerology が出るか）
- MRSS 評価の中間結果

## 5. CSI（Rel-19 AI/ML two-sided + 6GR CSI）

### 5.1 アジェンダ位置と背景

- **Rel-19 (NR_AIML_air_Ph2)**: WI 化 RAN#108（2025-06）。Case-0 = **UE 側 encoder + gNB 側 decoder の two-sided AI モデル**による CSI 圧縮（Type-II 派生）。
- **6GR**: CSI は Multi-antenna umbrella の必須サブアジェンダ。256+ CSI port 想定で「CSI accuracy が制限因子」。

### 5.2 #124 → #124bis の進展

| 観点 | 内容 |
|:---|:---|
| **#124 で議論された設計面** | (a) 標準化されたデータセット形式、(b) **ベンダー間の training collaboration**（pairing / dataset exchange）、(c) AI/ML proposal は **non-AI/ML 並列評価** が義務 |
| **#124bis のフォーカス** | inter-vendor pairing / model exchange のメカニズム具体化 |
| **既知の性能** | CSI feedback overhead 削減 **8〜79%**、user throughput **0〜17%** 増（複数社シミュレーション平均） |
| **6GR CSI** | 256〜512+ CSI port のスケーラブル設計、SRS 仕様の UE 要件強化が議論の中心 |

### 5.3 何が問題視されているか

| 観点 | Pro | Con |
|:---|:---|:---|
| **two-sided AI** | 大幅なオーバーヘッド削減、Type-II 性能を超える可能性 | **inter-vendor 互換**（model pairing / training data 共有）が最大の難題 |
| **JSCM（JSCC CSI）** | Samsung 提案の Joint Source-Channel mapping、フィードバックチェーン全体の最適化 | 標準化スコープが広く、検証負荷大 |
| **大規模 CSI port** | 6G ピーク性能の前提条件 | UE/RS 設計が爆発的に複雑化 |

### 5.4 #125 の見どころ

- two-sided 互換性のための **データセット / pairing API** の合意
- 6GR CSI のサブ AI（CSI-RS / SRS / フィードバック構造）の初期方向性

> **詳細別ノート**: [[260427_RAN1-124bis-DL-CSI-BM-AIML-調査]]

## 6. Control（PDCCH 物理層 / DCI, 6GR 10.5.2.1）

### 6.1 アジェンダ位置と背景

- **AI 10.5.2.1**: 「**DL 制御チャネルをどこ・どのように送るか**」（"how to send"）。
- **AI 10.5.4.1**（Scheduling, §7 参照）: 「**何を送るか**（DCI 内容、スケジューリング決定）」。本 split は #124bis Note 2 で明示。
- 5G NR の CORESET / Search space / Polar(Aggregated DCI) を継承。

### 6.2 #124 → #124bis の進展

| 観点 | 内容 |
|:---|:---|
| **論点** | (a) Aggregation level の柔軟化、(b) UE 共通 vs UE 固有 search space の構造、(c) D-MIMO 環境での共通制御送信、(d) blind decoding 数の制約 |
| **二次情報の限界** | Apex Standards が「control/HARQ provisions が体系的サブアジェンダにある」と要約のみ |
| **DCI overhead vs アンテナ数** | CSI port 増 = reference signal 増 = control overhead 圧迫、容量損失 5% 以下が業界目標 |

### 6.3 何が問題視されているか

- **PDCCH の信頼性 vs 効率**: 6G では URLLC / XR の遅延要件が厳しく、blind decoding 数を増やすか、結合検出を導入するか
- **大規模アンテナ時代の制御チャネル**: ビーム fail / track loss 時のフォールバック設計が未確定
- **D-MIMO 親和**: 複数 TRP からの同一 DCI 送信時の組合せ規則

### 6.4 #125 の見どころ

- Search space の **粒度** に対する初期合意
- DCI フォーマットの **エントロピー削減**（共通フィールド統合）

## 7. Scheduling（6GR 10.5.4.1）

### 7.1 アジェンダ位置と背景

- **AI 10.5.4.1**: PDCCH の **「何を送るか」** を扱う。DCI 内容、スケジューリング決定ロジック、リソース割当規則。
- 5G の K0/K1/K2、TBS テーブル、frequency hopping を継承。

### 7.2 #124 → #124bis の進展

| 観点 | 内容 |
|:---|:---|
| **論点** | (a) スモールデータ高速接続（"96% of sessions are small" — [arXiv 2508.08225](https://arxiv.org/html/2508.08225v1)）、(b) **Early CSI reporting** で initial scheduling 高速化、(c) Configured Grant の AI 連動 |
| **6G-6G aggregation 議論** | RP#111 で RAN1 タスク化された 6G-6G スペクトルアグリゲーション（特に非コロケート、バックホール遅延 ≤10ms 想定）の物理層・スケジューリング影響を 2026-09 までに評価 |

### 7.3 何が問題視されているか

- **バースト性トラフィック**: 実ネットワーク観察上、サンプル数では小セッションが多数だが容量はごく少数の大セッションが食う。スケジューラはこのスキューに最適化される必要
- **MRSS とスケジューリング**: 5G/6G 同時運用時のリソース調停
- **AI 親和**: RAN1 が踏み込めるのか、O-RAN (RIC) に委ねるのか

### 7.4 #125 の見どころ

- 6G-6G aggregation の物理層 / スケジューリング影響評価の **中間結果**
- Configured Grant の 6G 拡張可否

## 8. HARQ（6GR / Rel-20）

### 8.1 アジェンダ位置と背景

- **6GR 10.5 系の HARQ サブアジェンダ**（control/HARQ provisions として束ねられる）。
- 5G の 16 process、CBG retransmission、A/N codebook を継承前提。

### 8.2 #124 → #124bis の進展

| 観点 | 内容 |
|:---|:---|
| **論点（公開情報）** | (a) NTN（高遅延）での HARQ disable / number of process 拡張、(b) AI 連動の retransmission 戦略、(c) URLLC 向け超低遅延 A/N |
| **二次情報の限界** | "control/HARQ provisions" が「sub-agenda として体系化されている」レベルの要約のみ。**個別合意は要一次情報** |

### 8.3 何が問題視されているか

- **NTN 統合**: GEO 衛星では RTT が大きく HARQ を諦めるか、process 数を爆発させる必要
- **超低遅延 A/N**: PDCCH 直後の A/N feedback タイミング短縮
- **A-IoT との非対称性**: 受動デバイスでは古典的 HARQ が成立せず、ARQ も含む再設計

### 8.4 #125 の見どころ

- HARQ process 数 / A/N timing の **新規格上限**
- TN/NTN 共通フレームワークでの HARQ 動作モード

> **詳細別ノート**: [[260501_TN-NTN共通HARQ_変遷]]（#124 で TN performance prioritized 原則確定、#124bis で HARQ stalling 4 解決策 + TN harmonization の FFS 化、AI 番号 10.5.4.3 / 10.7 を agenda.csv で確定）

## 9. WUS（LP-WUS, Rel-19 完了 → Rel-20 拡張）

### 9.1 アジェンダ位置と背景

- **Rel-19 LP_WUS WI**: RAN1 機能フリーズ 2025-06 で完了。仕様化フェーズ。
- LP-WUS（Low Power Wake-Up Signal）+ LP-SS（Low Power Synchronization Signal）の対設計。

### 9.2 設計概要（[arXiv 2507.10207](https://arxiv.org/html/2507.10207v1)）

| 項目 | 仕様 |
|:---|:---|
| **占有帯域** | 12 RB（4.32 MHz）@ 20 MHz channel |
| **変調** | **MC-OOK**（Multi-Carrier On-Off Keying） — OFDM 上の OOK エミュレーション |
| **受信機** | エンベロープ検出器（ED）でメインラジオ OFF のまま検出可 |
| **電力削減** | 従来 5G paging に対し **最大 80%** |

### 9.3 #124 → #124bis での扱い

| 観点 | 内容 |
|:---|:---|
| **位置づけ** | Rel-19 maintenance（細部の corrections のみ） |
| **Rel-20 進化** | A-IoT との統合議論、低消費電力受信機の精度向上、複数 UE 多重 |
| **RAN4 連携** | 受信機実装要件 / RF 性能の試験仕様化 |

### 9.4 何が問題視されているか

| 観点 | Pro | Con |
|:---|:---|:---|
| **MC-OOK** | OFDM ベースで gNB 送信器が再利用可、UE 側は単純な ED で OK | 周波数選択性チャネルでの感度低下 |
| **狭帯域 12 RB** | 4.32 MHz と狭く LP-SS と組合せても電力少 | 割当の柔軟性が低く、運用上の周波数再利用に制約 |

### 9.5 #125 の見どころ

- Rel-19 仕様確定の **最終確認** が中心、新規論点は限定的見込み
- Rel-20 の更なる省電力モード（"LP-WUS phase 2"）の SI 立上げ機運

## 10. NTN（Non-Terrestrial Networks）

### 10.1 アジェンダ位置と背景

- **Rel-19**: NR-NTN Phase 3 + IoT-NTN（**Store-and-Forward**, regenerative payload）。RAN1 機能フリーズ完了。
- **Rel-20**: NTN Phase 4（拡張）として継続。
- **6GR**: TN/NTN 共通フレームワーク設計が SI スコープ。

### 10.2 #124 → #124bis の進展

| 観点 | 内容 |
|:---|:---|
| **#124 合意（Ofinno）** | 「**TN 性能を優先する**（terrestrial-network performance is prioritized when designing common TN/NTN solutions）」 — 共通設計時の優先順位原則を明文化 |
| **評価フレームワーク** | 周波数帯と衛星軌道（LEO / MEO / GEO）の組合せが評価対象として固まる |
| **業界実装** | 2026-01 CES で **Keysight × Samsung が NR-NTN Band n252 (Rel-19) の end-to-end** デモ。SAT-to-SAT モビリティ・cross-vendor IOP 検証 |
| **Store-and-Forward** | IoT-NTN で regenerative payload 上の S&F 動作 + feeder link switchover が設計対象 |

### 10.3 何が問題視されているか

| 観点 | Pro | Con |
|:---|:---|:---|
| **TN/NTN 共通設計** | 端末ベンダーの設計コスト削減 | TN 優先により NTN 固有最適化が劣後 |
| **regenerative payload** | 衛星上に gNB を置くことでバックホール減 / 地上ハンドオーバ減 | 衛星実装複雑度・電力 / SWaP 制約 |
| **direct-to-cell (D2C)** | 携帯電話そのものが衛星接続可能、市場拡大 | スペクトル共有・規制（n252 等）整理が前提 |

### 10.4 #125 の見どころ

- TN/NTN 共通 IA / HARQ の合意点 vs 分離点
- Phase 4 のスコープ確定

> **詳細別ノート**: [[260501_TN-NTN共通HARQ_変遷]]（6GR NTN AI 10.7 / Rel-19 NTN maintenance AI 8.7→8.6 を agenda.csv で確定。HARQ stalling Sol.1〜4 と TN harmonization FFS、MEO-7000 追加、k_offset broadcast 採用）

## 11. ISAC（Integrated Sensing and Communication）

### 11.1 アジェンダ位置と背景

- **Rel-19**: ISAC channel modeling SI（RAN1#116（2024-02）開始 → RAN1#121（2025-05）終了）。出力は **TR 38.901 拡張**として RAN#108（2025-06）承認。
- **継続 SI**: gNB **mono-static sensing for UAV ユースケース** の性能評価 + RAN/CN シグナリング。SA2 と協調。

### 11.2 #124 → #124bis での扱い

| 観点 | 内容 |
|:---|:---|
| **位置づけ** | channel modeling SI は完了済。**実用シナリオ評価フェーズ** |
| **対象ターゲット** | UAV、屋内人、屋外人、自動車、AGV、道路 / 線路上の障害物（#116 で承認済リスト） |
| **進捗** | 124bis では SA2 連携した RAN/CN シグナリング設計の議論。**具体合意の二次情報は未確認** |

### 11.3 何が問題視されているか

| 観点 | Pro | Con |
|:---|:---|:---|
| **mono-static gNB sensing** | 既存基地局の二次利用、追加インフラ不要 | プライバシー、UAV 検出感度、誤検出率 |
| **target / background channel 分離** | モデルが解析的に扱いやすく標準化向き | 実フィールドでの検証データ不足 |
| **波形再利用** | 通信用 OFDM/SC をそのまま sensing に使う | 専用 sensing waveform より感度劣化 |

### 11.4 #125 の見どころ

- mono-static sensing の **具体評価メトリクス**（detection probability、range/velocity 解像度）合意
- bi-static / multi-static への拡張スコープ

## 12. A-IoT（Ambient IoT）

### 12.1 アジェンダ位置と背景

- **Rel-19**: A-IoT SI（RAN1 完了）→ WI 化。**Device 1**（~1 µW peak power）を標準化対象。
- **Rel-20**: Ambient IoT 進化（device 2a / 2b、より高機能）が議論対象。

### 12.2 Device 1 仕様（[arXiv 2501.08990](https://arxiv.org/pdf/2501.08990)）

| 項目 | 仕様 |
|:---|:---|
| **ピーク電力** | ~1 µW |
| **エネルギー** | 内部蓄電（capacitor 等） |
| **受信機** | RF envelope detector |
| **同期精度** | initial SFO ≤ 105 ppm |
| **増幅** | R2D / D2R いずれも device 内増幅なし（passive / semi-passive） |

### 12.3 #124 → #124bis での扱い

| 観点 | 内容 |
|:---|:---|
| **位置づけ** | Rel-19 maintenance + Rel-20 拡張議論 |
| **Rel-20** | Coverage Enhancement Phase 3 と並ぶ拡張枠で「Rel-19 デザインの evolution」として定義 |
| **NTN との交差** | NTN 経由の A-IoT バックホールが新たな関心領域 |

### 12.4 何が問題視されているか

| 観点 | Pro | Con |
|:---|:---|:---|
| **Device 1（passive）** | 電池不要、RFID 後継として大量展開可 | 通信距離・データレートが極端に低い |
| **MC-OOK（LP-WUS と共通）** | 受信機共通化で BOM 削減 | 電力予算がさらに厳しい A-IoT で十分か未検証 |
| **3GPP エコシステム化** | 標準スペック化で IoT 市場が急拡大 | バッテリーレス前提のサプライチェーン未成熟 |

### 12.5 #125 の見どころ

- Rel-20 A-IoT の **device 2 クラス**（半受動 / 能動）への拡張範囲
- NTN-A-IoT 統合の SI 立上げ機運

## 13. 横断観察（What this means for SEP search）

### 13.1 議論パターンの共通項

1. **「5G 継承 + 限定的革新」** — Waveform / Channel coding / MIMO は 5G ベースライン継承で大枠合意、革新は性能実証の上で初めて検討
2. **AI/ML が全層に滲透するが標準化スコープは狭い** — CSI / Beam Mgmt / WUS が Rel-19 で先行、6GR では AI/ML proposal は「**non-AI/ML 並列評価が義務**」という慎重原則
3. **MRSS（5G-6G 共存）が制約条件** — 容量損失 <5% 縛りが多くのトピックの設計範囲を狭めている
4. **NTN / A-IoT が多くのトピックの「制約源」** — HARQ・IA・Waveform で NTN 親和性が要件化

### 13.2 特許の進歩性が出やすい領域（仮説、要深堀）

| 領域 | 仮説的着想ポイント |
|:---|:---|
| **CSI inter-vendor pairing** | データセット交換 / model pairing API の具体化はベンダー固有の最適化と摩擦を生み、進歩性の余地大 |
| **Multi-rank UL DFT-s-OFDM** | 高 PAPR を抑えつつ rank-2+ を実現する FDSS 派生（5G では rank-1 中心） |
| **TN/NTN 共通 HARQ** | 低 RTT の TN と高 RTT の NTN を同一 process 構造で扱う規則の発明余地 |
| **A-IoT × LP-WUS 共通受信機** | エネルギーバジェット <1 µW での MC-OOK 復号最適化 |
| **D-MIMO 標準化障壁の回避** | 位相同期要件を緩めた cooperative beamforming のシグナリング |
