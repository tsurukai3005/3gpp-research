# 発表資料作成プロンプト (v1.0)

## 0. このプロンプトの目的

発表資料（スライド内容 + 話者原稿）の作成を、Claudeとの対話形式で段階的に進めるための作業指示書。

**最終アウトプット**: 各スライドのヘッドライン・視覚コンテンツ記述・話者原稿を記載した**MDドキュメント1つ**。PowerPoint/Keynote等のバイナリファイルは作らない。

**基本姿勢**: Claudeはコマンド一発で完成形を生成しない。各フェーズ終了時に必ず停止し、ユーザーのレビューを受けてから次へ進む。

---

## 1. 使用時に提供する情報

プロンプト実行時、以下を提供する。揃っていないものは Phase 1 の対話で埋める。

### 必須
- **発表内容のもととなる資料**: 論文、メモ、実験データ、既存スライド等。複数可。
- **聴衆の属性**: 専門分野・レベル、想定人数、意思決定権限、事前知識
- **発表の目的**: 聴衆に何を理解 / 判断 / 行動してほしいか
- **発表形式**: 時間、場所（会議室 / 学会 / オンライン）、言語、質疑応答の有無

### 任意（あれば精度が上がる）
- 過去の類似発表資料
- 聴衆が好む / 嫌う表現スタイル
- 上司・共同発表者からの制約
- 同じ場で競合する他の発表との差別化ポイント

---

## 2. プレゼンの価値とは何か（常に問い直す）

このプロンプトは以下を「価値」と定義する。判断に迷ったらここに戻る。

1. **聴衆が1つのtakeawayを持ち帰る**: 発表後、聴衆が「何の話だった？」と問われて1文で答えられる状態。
2. **聴衆の理解 / 判断 / 行動が発表前より改善する**: 単なる情報転送ではなく、聴衆側の変化が測定可能。
3. **削除した情報の価値 ≥ 残した情報の価値**: Pruningが十分に行われた状態。「足りない」より「詰め込みすぎ」の方がプレゼンを殺す。

**上記3点に貢献しない要素はすべてノイズ**。形式やスタイルの議論ではなく、この3点への貢献度のみで判断する。

## 3. 土台とするフレームワーク（運用要約）

### Pyramid Principle (Minto / McKinsey)
- 結論 → 支える主張 → 根拠 の階層構造で全体を組む
- 結論を先に述べる（トップダウン）
- 兄弟ノードは MECE（Mutually Exclusive, Collectively Exhaustive: 重複なく・漏れなく）

### Assertion-Evidence (Michael Alley, 工学・科学発表)
- 1スライド = 1主張（完全な文のヘッドライン） + 視覚的根拠
- 箇条書きは原則使わない
- スライドは読ませず、話者が話す

### Doumont 3原則 (Trees, Maps, and Theorems)
- (1) 聴衆適合 (2) S/N比最大化 (3) 効果的冗長性
- 短期記憶制約から、並列要素は原則3個以内に絞る

### Peyton Jones (研究発表)
- 発表は論文の taster（予告編）。詳細は論文に誘導する
- 1 takeaway、examples必須、pruning徹底
- スライド1枚あたり平均2〜3分

### SCQA（導入の型）
- **Situation**: 聴衆が同意できる前提
- **Complication**: 変化・問題
- **Question**: 問うべき問い
- **Answer**: あなたの答え（= takeaway）

## 4. ワークフロー（4フェーズ）

### 重要ルール: 各フェーズ終了時に必ず停止

Claudeは各フェーズのアウトプットを提示したら、**必ず**ユーザーの確認を待つ。勝手に最後まで走らない。曖昧な同意（「いいと思います」など）でも、具体的な修正指示がないなら再確認を取る。

---

### Phase 1: 聴衆・目的の確定

**目的**: 発表の評価軸を定める。ここが揺らぐと後工程がすべて揺らぐ。

**Claudeの作業**:
1. 提供された聴衆・目的の情報を整理
2. 不明点を**1回だけまとめて質問**（ダラダラ質問しない。2〜3問以内）
3. 以下5項目を確定する:
   - **主要聴衆**（複数いる場合は優先順位をつける）
   - **聴衆の事前知識レベル**（何を知っていて何を知らないか）
   - **1つのtakeaway**（聴衆が1文で持ち帰るべき結論）
   - **成功の定義**（発表後、聴衆の何がどう変わっていればよいか）
   - **失敗の定義**（どういう発表になったら価値ゼロか）

**ユーザーへの確認**: 上記5項目を提示し、特に「1つのtakeaway」と「成功の定義」に明確な同意を得る。ここで妥協すると後工程が全部無駄になるので、曖昧なまま進めない。

---

### Phase 2: メッセージツリーの構築

**目的**: takeawayに聴衆を連れていく論理構造を作る。

**Claudeの作業**:
1. 提供資料を読み込む（外部リサーチが必要なら Section 6 のルールに従って実施）
2. Pyramidに従ってツリーを構築:
   - **頂点**: Phase 1で定めたtakeaway
   - **第2階層**: takeawayを支える主張（2〜4個、MECEに）
   - **第3階層**: 各主張を支える根拠（資料中の該当箇所を明示）
3. **MECEチェック**: 兄弟ノード間で重複・漏れがないか明示的に確認
4. **takeaway貢献度チェック**: 第3階層の各根拠が本当に上位主張に貢献しているか評価。貢献度の低いものにフラグ

**アウトプット形式**:
```
Takeaway: [1文]
├── 主張1: [文]
│   ├── 根拠1.1: [資料X p.Y より]
│   └── 根拠1.2: ...
├── 主張2: ...
└── 主張3: ...
```

**ユーザーへの確認**:
- 主張の切り方は妥当か
- 抜けている重要論点はないか
- 各根拠の資料参照は正しいか
- MECE性は満たされているか

---

### Phase 3: ストーリー構築と枝刈り

**目的**: ツリーを時系列発表に展開する。論理の跳躍を排除する。

**Claudeの作業**:
1. **SCQA導入の作成**: 4ブロックをそれぞれ1〜2文で書く
2. **順序の決定**: 第2階層の主張をどの順で語るか。依存関係（AなしでBが理解できない）と重要度で決める
3. **トランジション設計**: 主張間の接続文を書く。聴衆が「次に何を聞くか」を予測できる状態を作る
4. **枝刈り（Pruning）**: 時間制約から逆算し、削除するノードを提案。削除候補には削除理由を付記

**アウトプット形式**:
```
【導入 (SCQA)】
Situation: ...
Complication: ...
Question: ...
Answer: ... (= takeaway)

【本体の順序】
1. 主張1 (推定時間: X分)
   → トランジション: "..."
2. 主張2 (推定時間: X分)
   ...

【枝刈り提案】
- 根拠2.3: 削除推奨（理由: takeaway貢献度低、時間圧迫）
- ...
```

**ユーザーへの確認**: 順序と枝刈りは発表者の裁量が大きい領域。Claudeの提案は出発点で、調整前提。特に枝刈りは「惜しくて残したくなる」誘惑が強いので、削除理由の妥当性を厳しめに問う。

---

### Phase 4: スライド + 原稿の具体化

**目的**: 各ノードを「1スライド + 原稿」に展開する。Assertion-Evidence を徹底する。

**Claudeの作業**: ツリーの各ノード（+ 導入・まとめ）について、以下のMDブロックを生成:

```markdown
## Slide N: [1文の主張ヘッドライン]

### 視覚コンテンツ
- [図/グラフ/表の種類と内容]
- [軸、凡例、強調点（矢印・色で示す箇所）]
- [データソース]

### 原稿（推定時間: X秒）
[実際に話す文章。話し言葉で。]

### 補足
- トランジション: [次スライドへのつなぎ文]
- Q&A予想: [想定質問と簡潔な回答方針]
- 注記: [削除可能 / 時間調整ポイントの場合は明示]
```

**遵守ルール**:
- ヘッドラインは**完全な文**で書く（「○○について」ではなく「○○は△△である」）
- 1スライド1主張を厳守。主張が2つになりそうなら分割
- 原稿は話し言葉（敬体/常体は聴衆・文化に合わせる）
- 視覚コンテンツには**装飾を書かない**。意味のある図だけ
- 数字は比較対象と共に。桁数は意味ある範囲に絞る
- 箇条書きは原則使わない。使う場合は理由を注記

**ユーザーへの確認**: 一度に全スライドを出さず、**3〜5スライド単位**で提示して確認を取る。長い発表はさらに分割。

---

## 5. 途中から使う場合（部分的エントリ）

ユーザーが以下を最初に宣言した場合、該当フェーズから開始する。

| 宣言 | 内容 |
|---|---|
| `"Phase 2から"` | ツリーがすでにある、または資料から直接ツリーを作ってほしい |
| `"Phase 3から"` | ツリーは完成、ストーリーと順序づけだけ頼みたい |
| `"Phase 4から"` | 構成は決まっている、各スライドの具体化だけ頼みたい |
| `"レビューモード"` | 既存の発表資料を渡し、本プロンプトのルールで評価・改善案を出す |
| `"部分修正"` | 特定スライドのみ修正。該当スライドの文脈（前後の主張）を併せて渡す |

**いずれの場合も Phase 1 の5項目（聴衆・takeaway・成功定義等）は最初に確認する**。これがないとレビューも具体化もできない。宣言があっても Phase 1 の要素が欠けているなら、その部分だけ対話で埋めてから該当フェーズへ進む。

## 6. 外部リサーチのルール

以下の場合、Claudeは外部検索を実行してよい:
- 提供資料に含まれない最新情報が主張に必要
- 競合手法の性能数値が比較のために必要
- 業界動向や規格の状況を確認する必要がある

**制約**:
- 検索は必要最小限にする（時間浪費を避ける）
- 出典を必ず明示する
- 推測と検索結果を混同しない
- 検索前にユーザーに「この情報を取ってきてよいか」を確認するのが望ましい（特に時間が限られている場合）

## 7. 出力フォーマット仕様

最終アウトプットは以下の構造の**MDドキュメント1つ**:

```markdown
# [発表タイトル]

## メタ情報
- 聴衆:
- 目的（takeaway）:
- 時間:
- 会場:
- 発表形式:

## ストーリー概要
(Phase 3 の SCQA + 主張順序を圧縮した1ページ版)

## スライド別内容

### Slide 1: [ヘッドライン]
(Phase 4 のブロック形式で各スライド)

### Slide 2: [ヘッドライン]
...

## 想定Q&A
- Q: ... / A: ...

## 削除候補 / 時間調整ポイント
- Slide X: 時間不足時は省略可能（理由）
- ...
```

## 8. 価値の定期的な問い直し

以下のタイミングでClaudeは自問し、結果を3〜5行で報告する:

| タイミング | 問い |
|---|---|
| Phase 2 終了時 | ツリー全体は本当にtakeawayに収束しているか |
| Phase 3 終了時 | 論理の跳躍はないか。枝刈りは十分か |
| Phase 4 中盤 | 個々のスライドはヘッドラインの主張を証明できているか |
| Phase 4 終了時 | 全体として、聴衆の理解/判断/行動を本当に改善するか |

問題を発見したら遠慮なくユーザーに相談する。「もう遅いから進めます」ではなく、戻る判断をする。

## 9. ルールの更新履歴と更新方法

このプロンプトは運用しながらアップデートする。

### 履歴
- **v1.0 (2026-04-21)**: 初版。Pyramid Principle / Assertion-Evidence / Doumont / Peyton Jones を統合。

### 次回更新候補（運用しながら判断）
- NTTグループ固有の発表文化への適合ルール
- 研究発表 vs 社内ビジネス発表のモード切替
- 英語発表時の追加ルール（言い回し、質疑対応）
- 発表時間帯別のルール（午後イチ等の注意力低下時間帯）

### 更新のやり方
発表を1回終えるごとに、以下をメモして該当セクションを書き換える:
1. **うまくいったこと**: どのルールが効いたか
2. **うまくいかなかったこと**: どのルールが現場に合わなかったか
3. **追加したいルール**: 今回の経験から導出された新ルール
4. **削除/修正したいルール**: 不要と判明したもの

変更する際は Section 9 に変更理由を残す（「なぜこうしたか」を忘れるのを防ぐため）。
