# build-presentation

## モチベーション

研究調査の成果を発表に変換する際、情報を詰め込みすぎて聴衆に伝わらない問題が起きる。
Pyramid Principle / Assertion-Evidence / SCQA を体系的に適用することで、
**1つの takeaway を確実に伝える発表資料**を作る。

発表は論文の taster（予告編）。詳細は論文・ノートに誘導し、スライドでは削ぎ落としを徹底する。

## 使い方

```
/build-presentation <ソース>
/build-presentation <ソース> --phase <1|2|3|4|review|patch>
/build-presentation <ソース> --lang <ja|en>
```

- `<ソース>`: `documents/` 配下のノート、論文メモ、または自由テキスト
- `--phase`: 途中フェーズから開始（デフォルト: 1）
- `--lang`: 出力言語（デフォルト: ja）

### 途中エントリ

| 指定 | 用途 |
|:---|:---|
| `--phase 2` | 資料からツリーを直接構築したい |
| `--phase 3` | ツリーは完成済み、ストーリーと順序づけだけ頼みたい |
| `--phase 4` | 構成は決まっている、各スライドの具体化だけ頼みたい |
| `--phase review` | 既存の発表資料を渡し、価値の定義に照らして評価・改善案を出す |
| `--phase patch` | 特定スライドのみ修正（前後の主張の文脈を併せて提供すること） |

**いずれの場合も Phase 1 の5項目（聴衆・takeaway・成功定義等）は最初に確認する。**
宣言があっても5項目が欠けているなら、その部分だけ対話で埋めてから該当フェーズへ進む。

## 入力と前提

- **必須入力**: 発表のもとになる資料（リポジトリ内ノートのパス、テキスト、またはファイル添付）
- **最低限の情報**: 以下のうち最低2つ（残りは Phase 1 の対話で埋める）
  - 聴衆の属性
  - 発表の目的
  - 発表時間
- **任意（あれば精度が上がる）**:
  - 過去の類似発表資料
  - 聴衆が好む/嫌う表現スタイル
  - 上司・共同発表者からの制約
  - 同じ場で競合する他の発表との差別化ポイント
- **実行不可の条件**: ソース資料がゼロの場合 → 先に `/survey-topic` や `/digest-paper` を案内する

## 実行フロー

### 重要ルール

1. **各フェーズ終了時に必ず停止し、ユーザーの確認を得てから次へ進む。**
   曖昧な同意（「いいと思います」等）でも具体的修正指示がなければ再確認する。
2. **問題を発見したらフェーズを巻き戻す。**
   「もう遅いから進めます」ではなく、戻る判断をする。Phase 3 で takeaway の弱さに気づいたら Phase 1 に戻る。
3. **既存の成果物ファイル**（`documents/` 配下）がある場合は読み込んで継続する。

---

### Phase 1: 聴衆・目的の確定

**目的**: 発表の評価軸を定める。ここが揺らぐと後工程がすべて揺らぐ。

1. ソース資料を読み込む
2. 不明点を **2〜3問以内** にまとめて質問する（ダラダラ聞かない）
3. 以下5項目を確定する:
   - **主要聴衆**（複数いる場合は優先順位付き）
   - **聴衆の事前知識レベル**（何を知っていて何を知らないか）
   - **1つの takeaway**（聴衆が1文で持ち帰る結論）
   - **成功の定義**（発表後、聴衆の何がどう変わればよいか）
   - **失敗の定義**（どうなったら価値ゼロか）
4. 5項目を提示し、特に takeaway と成功定義に明確な同意を得る。ここで妥協すると後工程が全部無駄になるので、曖昧なまま進めない

---

### Phase 2: メッセージツリーの構築

**目的**: takeaway に聴衆を連れていく論理構造を作る。ツリーなしにスライドを書き始めると、「言いたいことの羅列」になり takeaway に収束しない。

1. ソース資料から Pyramid 構造でツリーを構築:
   - **頂点**: Phase 1 の takeaway
   - **第2階層**: takeaway を支える主張（2〜4個、MECE）
   - **第3階層**: 各主張の根拠（資料の該当箇所を明示）
2. **前提概念の洗い出し**: 説明対象セクション外だが理解に必要な前提知識を列挙する。ソースノートのスコープに閉じず、聴衆が各主張を理解するために必要な概念が全てツリーに含まれているか確認する
3. **サブセクション網羅チェック**: 対象セクションの全サブセクションに対し、説明の有無を一覧で確認する。意図的に省略する場合はその理由を明記する
4. MECE チェック: 兄弟ノード間の重複・漏れを確認
5. takeaway 貢献度チェック: 貢献度の低い根拠にフラグ

**アウトプット形式**:
```
Takeaway: [1文]
├── 主張1: [文]
│   ├── 根拠1.1: [資料X p.Y より]
│   └── 根拠1.2: ...
├── 主張2: ...
└── 主張3: ...
```

**ユーザーへの確認**（以下を明示的に問う）:
- 主張の切り方は妥当か
- 抜けている重要論点はないか
- 各根拠の資料参照は正しいか
- MECE 性は満たされているか

**自問**: ツリー全体は本当に takeaway に収束しているか？（3〜5行で報告。問題を発見したら Phase 1 に戻ることを提案する）

---

### Phase 3: ストーリー構築と枝刈り

**目的**: ツリーを時系列の発表に展開する。論理の跳躍を排除し、聴衆が迷子にならない流れを作る。

1. **SCQA 導入を作成**: 4ブロックをそれぞれ1〜2文で書く
   - Situation: 聴衆が同意できる前提
   - Complication: 変化・問題
   - Question: 問うべき問い
   - Answer: あなたの答え（= takeaway）
2. **順序の決定**: 第2階層の主張をどの順で語るか。依存関係（A なしで B が理解できない）と重要度で決める
3. **トランジション設計**: 主張間の接続文を書く。聴衆が「次に何を聞くか」を予測できる状態を作る
4. **時間配分**: 発表時間から逆算し、各主張の配分を決定する（目安: 1スライドあたり2〜3分。内容と聴衆に応じて調整）
5. **枝刈り（Pruning）**: 時間制約から逆算し、削除ノードを提案（削除理由付き）。枝刈りは「惜しくて残したくなる」誘惑が強いので、削除理由の妥当性を厳しめに問う

**アウトプット形式**:
```
【導入 (SCQA)】
Situation: ...
Complication: ...
Question: ...
Answer: ... (= takeaway)

【本体の順序】
1. 主張1 (推定時間: X分)
   → トランジション: "..."
2. 主張2 (推定時間: X分)
   ...

【枝刈り提案】
- 根拠2.3: 削除推奨（理由: takeaway貢献度低、時間圧迫）
- ...
```

**ユーザーへの確認**: 順序と枝刈りは発表者の裁量が大きい領域。Claude の提案は出発点で、調整前提。特に枝刈りは「惜しくて残したくなる」誘惑が強いので、削除理由の妥当性を厳しめに問う。

**自問**: 論理の跳躍はないか。枝刈りは十分か？（3〜5行で報告。問題を発見したら前のフェーズに戻ることを提案する）

---

### Phase 4: スライド + 原稿の具体化

**目的**: 各ノードを「1スライド + 原稿」に展開する。Assertion-Evidence を徹底する。

各ノードを以下の形式で展開する。**3〜5スライド単位** で提示し確認を取る（一度に全スライドを出すと、ユーザーがレビューしきれず品質が下がるため）。長い発表はさらに分割する。

```markdown
## Slide N: [1文の主張ヘッドライン]

### 視覚コンテンツ
- [図/グラフ/表の種類と内容]
- [軸、凡例、強調点（矢印・色で示す箇所）]
- [データソース]

### 原稿（推定時間: X秒）
[話し言葉で書く]

### 補足
- トランジション: [次スライドへのつなぎ]
- Q&A予想: [想定質問と回答方針]
- 注記: [削除可能/時間調整ポイントの場合は明示]
```

**遵守ルール**:
- ヘッドラインは **完全な文**（「○○について」→「○○は△△である」）
- 1スライド = 1主張（2つになりそうなら分割）
- **原稿は箇条書きで要素を簡潔に並べる**（口語表現は発表者がその場で補足する前提）
- 視覚コンテンツは意味のある図のみ（装飾を書かない）
- 数字は比較対象と共に提示（桁数は意味ある範囲に絞る）
- **スライドタイトルは事実ベース**。比喩・強調表現を避ける。「スコープの宣言」と「事実の主張」を混同しない
- **計算式・用語の命名根拠を説明に含める**（例: 「なぜ minimum か」「なぜ common か」）
- 仕様セクション番号を対応情報として必ず明示する

**自問**: 個々のスライドはヘッドラインの主張を証明できているか？
全体として聴衆の理解/判断/行動を改善するか？（3〜5行で報告。問題を発見したら遠慮なくユーザーに相談する）

---

### レビューモード

`--phase review` で起動。既存の発表資料を渡し、以下の観点で評価する:

1. Section「価値の定義」に照らした評価
2. 各スライドが Assertion-Evidence を満たすか
3. 枝刈りの提案
4. SCQA の再構築案（必要なら）

### パッチモード

`--phase patch` で起動。特定スライドのみ修正する。

1. 対象スライドと前後の主張の文脈を受け取る
2. Phase 1 の5項目を確認（既存ファイルの frontmatter/メタ情報から取得可能）
3. 修正スライドが全体のストーリーライン・トランジションと整合するか検証
4. 修正結果を提示し、前後スライドへの影響があれば併せて報告

## 出力

- **形式**: チャット（対話形式で段階的に生成） + 最終ファイル
- **保存先**: `documents/<該当週フォルダ>/` 配下（リポジトリの命名規則に従う）
- **frontmatter**:

```yaml
---
title: "発表タイトル"
status: draft | reviewed | stable
created: YYYY-MM-DD
updated: YYYY-MM-DD
audience: "聴衆の属性"
takeaway: "1文の takeaway"
duration_min: N
venue: "会議室 / 学会 / オンライン"
format: "口頭発表 / ポスター / ライトニングトーク"
source_notes:
  - "ソースノートへの相対パス"
lang: ja | en
---
```

**最終ドキュメント構造**:
```markdown
# [発表タイトル]

## メタ情報
- 聴衆 / 目的 / 時間 / 会場 / 形式

## ストーリー概要
(SCQA + 主張順序の圧縮版)

## スライド別内容
### Slide 1: [ヘッドライン]
...

## 想定Q&A
- Q: ... / A: ...

## 削除候補 / 時間調整ポイント
- Slide X: 時間不足時は省略可能（理由）
```

## このスキル固有の注意点

### 価値の定義（常に問い直す）

1. **聴衆が1つの takeaway を持ち帰る**: 「何の話だった？」に1文で答えられる
2. **聴衆の理解/判断/行動が発表前より改善する**: 情報転送ではなく変化を起こす
3. **削除した情報の価値 ≥ 残した情報の価値**: 「詰め込みすぎ」は発表を殺す

**上記3点に貢献しない要素はすべてノイズ。** 形式やスタイルの議論ではなく、この3点への貢献度のみで判断する。

### フレームワーク（運用上の参照）

| フレームワーク | 要点 |
|:---|:---|
| Pyramid Principle | 結論→主張→根拠。トップダウン。兄弟は MECE |
| Assertion-Evidence | 1スライド=1主張（完全文ヘッドライン）+視覚的根拠 |
| Doumont 3原則 | 聴衆適合 / S/N比最大化 / 効果的冗長性。並列3個以内 |
| Peyton Jones | 発表は taster。1 takeaway、例必須、pruning 徹底。1枚あたり平均2〜3分 |
| SCQA | Situation → Complication → Question → Answer |

原典を読むとさらに理解が深まる（特に Alley と Doumont は工学発表者にとって実用的）:
- Barbara Minto, *The Pyramid Principle* (Pearson)
- Michael Alley, *The Craft of Scientific Presentations* (Springer)
- Jean-luc Doumont, *Trees, Maps, and Theorems* (Principiae)
- Simon Peyton Jones et al., "How to give a good research talk", *SIGPLAN Notices* 28(11), 1993

### 外部リサーチのルール

以下の場合のみ外部検索を実行する:
- 提供資料にない最新情報が主張に必要
- 競合手法の性能数値が比較に必要
- 業界動向・規格状況の確認が必要

制約: 最小限に留め、出典明示、推測と検索結果を混同しない。
時間が限られている場合は、検索前にユーザーに確認する。

### 3GPP研究コンテキストとの接続

- ソースとして `documents/` のノートを直接参照可能
- 技術的な根拠には原則1（出典記録）を適用する
- 実装制約ギャップ（原則2）は発表の差別化ポイントになりうる
- ノートの `axes:` 情報を活用して聴衆への文脈付けを行う

## 更新のしかた

発表を1回終えるごとに、以下をメモしてスキルを改善する:

1. **うまくいったこと**: どのルールが効いたか
2. **うまくいかなかったこと**: どのルールが現場に合わなかったか
3. **追加したいルール**: 今回の経験から導出された新ルール
4. **削除/修正したいルール**: 不要と判明したもの

### 次回更新候補（運用しながら判断）

- 研究発表 vs 社内ビジネス発表のモード切替
- 英語発表時の追加ルール（言い回し、質疑対応）
- 発表時間帯別のルール（午後イチ等の注意力低下時間帯）
