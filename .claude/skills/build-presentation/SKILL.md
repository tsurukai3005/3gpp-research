# build-presentation

## モチベーション

研究調査の成果を発表に変換する際、情報を詰め込みすぎて聴衆に伝わらない問題が起きる。
Pyramid Principle / Assertion-Evidence / SCQA を体系的に適用することで、
**伝えたいことを確実に伝える発表資料**を作る。

詳細は論文・ノートに誘導し、スライドでは削ぎ落としを徹底する。

## 使い方

```
/build-presentation <ソース>
/build-presentation <ソース> --phase <1|2|3|4|review|patch>
/build-presentation <ソース> --lang <ja|en>
```

- `<ソース>`: `documents/` 配下のノート、論文メモ、または自由テキスト
- `--phase`: 途中フェーズから開始（デフォルト: 1）
- `--lang`: 出力言語（デフォルト: ja）

### 途中エントリ

| 指定 | 用途 |
|:---|:---|
| `--phase 2` | 資料からツリーを直接構築したい |
| `--phase 3` | ツリーは完成済み、ストーリーと順序づけだけ頼みたい |
| `--phase 4` | 構成は決まっている、各スライドの具体化だけ頼みたい |
| `--phase review` | 既存の発表資料を渡し、価値の定義に照らして評価・改善案を出す |
| `--phase patch` | 特定スライドのみ修正（前後の主張の文脈を併せて提供すること） |

**いずれの場合も Phase 1 の5項目（聴衆・伝えたいこと・成功定義等）は最初に確認する。**
宣言があっても5項目が欠けているなら、その部分だけ対話で埋めてから該当フェーズへ進む。

## 入力と前提

- **必須入力**: 発表のもとになる資料（リポジトリ内ノートのパス、テキスト、またはファイル添付）
- **最低限の情報**: 以下のうち最低2つ（残りは Phase 1 の対話で埋める）
  - 聴衆の属性
  - 発表の目的
  - 発表時間
- **任意（あれば精度が上がる）**:
  - 過去の類似発表資料
  - 聴衆が好む/嫌う表現スタイル
  - 上司・共同発表者からの制約
  - 同じ場で競合する他の発表との差別化ポイント
- **実行不可の条件**: ソース資料がゼロの場合 → 先に `/survey-topic` や `/digest-paper` を案内する

## 実行フロー

### 重要ルール

1. **各フェーズ終了時に必ず停止し、ユーザーの確認を得てから次へ進む。**
   曖昧な同意（「いいと思います」等）でも具体的修正指示がなければ再確認する。
2. **問題を発見したらフェーズを巻き戻す。**
   「もう遅いから進めます」ではなく、戻る判断をする。Phase 3 で伝えたいことの弱さに気づいたら Phase 1 に戻る。
3. **既存の成果物ファイル**（`documents/` 配下）がある場合は読み込んで継続する。

---

### Phase 1: 聴衆・目的の確定

**目的**: 発表の評価軸を定める。ここが揺らぐと後工程がすべて揺らぐ。

1. ソース資料を読み込む
2. 不明点を **2〜3問以内** にまとめて質問する（ダラダラ聞かない）
3. 以下5項目を確定する:
   - **主要聴衆**（複数いる場合は優先順位付き）
   - **聴衆の事前知識レベル**（何を知っていて何を知らないか）
   - **伝えたいこと**（聴衆に伝えたい内容。少ない方が構造として望ましいが、複数でもよい）
   - **成功の定義**（発表後、聴衆の何がどう変わればよいか）
   - **失敗の定義**（どうなったら価値ゼロか）
4. 5項目を提示し、特に伝えたいことと成功定義に明確な同意を得る。ここで妥協すると後工程が全部無駄になるので、曖昧なまま進めない

---

### Phase 2: メッセージツリーの構築

**目的**: 発表の論理構造を作る。ツリーなしにスライドを書き始めると、「言いたいことの羅列」になり構造が崩れる。

1. **ツリー構造の決定**: Phase 1 の「伝えたいこと」に基づき、対話の中で妥当なツリー構造を決める
   - 伝えたいことが1つなら、それを頂点とする1本のピラミッド
   - 伝えたいことが複数あれば、それぞれを頂点とする複数のピラミッド
   - 各ピラミッドの構造: 頂点（伝えたいこと）→ 主張（2〜4個）→ 根拠（資料の該当箇所を明示）
   - ピラミッド間の関係（独立、順序依存、対比など）も明示する
2. **前提概念の洗い出し**: 説明対象セクション外だが理解に必要な前提知識を列挙する。ソースノートのスコープに閉じず、聴衆が各主張を理解するために必要な概念が全てツリーに含まれているか確認する
3. **サブセクション網羅チェック**: 対象セクションの全サブセクションに対し、説明の有無を一覧で確認する。意図的に省略する場合はその理由を明記する
4. MECE チェック: 兄弟ノード間の重複・漏れを確認
5. 重要度の判定: 各ノードの重要度（高/中/低）を判定し、理由を付記する

**アウトプット形式**（ピラミッドが複数の場合は繰り返す）:
```
伝えたいこと A: [文]
├── 主張1: [文] (重要度: 高)
│   ├── 根拠1.1: [資料X p.Y より]
│   └── 根拠1.2: ...
├── 主張2: ... (重要度: 中)
└── 主張3: ... (重要度: 高)

伝えたいこと B: [文]
├── ...

ピラミッド間の関係: A は B の前提知識となる
```

**ユーザーへの確認**（以下を明示的に問う）:
- ツリーの構造（ピラミッドの数と関係）は妥当か
- 主張の切り方は妥当か
- 抜けている重要論点はないか
- 各根拠の資料参照は正しいか
- 重要度の判定は妥当か

**自問**: ツリー全体は伝えたいことを過不足なく表現しているか？（3〜5行で報告。問題を発見したら Phase 1 に戻ることを提案する）

---

### Phase 3: ストーリー構築と枝刈り

**目的**: ツリーを時系列の発表に展開する。論理の跳躍を排除し、聴衆が迷子にならない流れを作る。

1. **SCQA 導入を作成**: 4ブロックをそれぞれ1〜2文で書く
   - Situation: 聴衆が同意できる前提
   - Complication: 変化・問題
   - Question: 問うべき問い
   - Answer: あなたの答え（= 伝えたいこと。複数の場合は要約）
2. **順序の決定**: ピラミッド間の順序、および各ピラミッド内の主張の順序を決める。依存関係（A なしで B が理解できない）と重要度で決める
3. **トランジション設計**: 主張間・ピラミッド間の接続文を書く。聴衆が「次に何を聞くか」を予測できる状態を作る
4. **枝刈り（Pruning）**: Phase 2 の重要度判定に基づき、削除ノードを提案（削除理由付き）。重要度が低く、かつ他のノードの理解に不要なものが削除候補。枝刈りは「惜しくて残したくなる」誘惑が強いので、削除理由の妥当性を厳しめに問う
5. **時間配分（参考）**: 発表時間が決まっている場合、各主張の大まかな配分を記載する。ただし正確な時間は読み上げるまで不明なので、あくまで目安

**アウトプット形式**:
```
【導入 (SCQA)】
Situation: ...
Complication: ...
Question: ...
Answer: ... (= 伝えたいこと)

【本体の順序】
1. 主張1 (重要度: 高)
   → トランジション: "..."
2. 主張2 (重要度: 中)
   ...

【枝刈り提案】
- 根拠2.3: 削除推奨（理由: 重要度低、他ノードの理解に不要）
- ...

【時間配分（参考）】
- 全体: 約N分（発表時間が決まっている場合）
- 各ブロックの大まかな比重
```

**ユーザーへの確認**: 順序と枝刈りは発表者の裁量が大きい領域。Claude の提案は出発点で、調整前提。

**自問**: 論理の跳躍はないか。重要度の高いノードが適切に配置されているか？（3〜5行で報告。問題を発見したら前のフェーズに戻ることを提案する）

---

### Phase 4: スライド + 原稿の具体化

**目的**: 各ノードを「1スライド + 原稿」に展開する。Assertion-Evidence を徹底する。

各ノードを以下の形式で展開する。**3〜5スライド単位** で提示し確認を取る（一度に全スライドを出すと、ユーザーがレビューしきれず品質が下がるため）。長い発表はさらに分割する。

```markdown
## Slide N: [1文の主張ヘッドライン]

### 視覚コンテンツ
- [図/グラフ/表の種類と内容]
- [軸、凡例、強調点（矢印・色で示す箇所）]
- [データソース]

### 原稿（推定時間: X秒）
[話し言葉で書く]

### 補足
- トランジション: [次スライドへのつなぎ]
- Q&A予想: [想定質問と回答方針]
- 注記: [削除可能/時間調整ポイントの場合は明示]
```

**遵守ルール**:
- ヘッドラインは **完全な文**（「○○について」→「○○は△△である」）
- 1スライド = 1主張（2つになりそうなら分割）
- **原稿は箇条書きで要素を簡潔に並べる**（口語表現は発表者がその場で補足する前提）
- 視覚コンテンツは意味のある図のみ（装飾を書かない）
- 数字は比較対象と共に提示（桁数は意味ある範囲に絞る）
- **スライドタイトルは事実ベース**。比喩・強調表現を避ける。「スコープの宣言」と「事実の主張」を混同しない
- **計算式・用語の命名根拠を説明に含める**（例: 「なぜ minimum か」「なぜ common か」）
- 仕様セクション番号を対応情報として必ず明示する

**自問**: 個々のスライドはヘッドラインの主張を証明できているか？
全体として聴衆の理解/判断/行動を改善するか？（3〜5行で報告。問題を発見したら遠慮なくユーザーに相談する）

---

### レビューモード

`--phase review` で起動。既存の発表資料を渡し、以下の観点で評価する:

1. Section「価値の定義」に照らした評価
2. 各スライドが Assertion-Evidence を満たすか
3. 枝刈りの提案
4. SCQA の再構築案（必要なら）

### パッチモード

`--phase patch` で起動。特定スライドのみ修正する。

1. 対象スライドと前後の主張の文脈を受け取る
2. Phase 1 の5項目を確認（既存ファイルの frontmatter/メタ情報から取得可能）
3. 修正スライドが全体のストーリーライン・トランジションと整合するか検証
4. 修正結果を提示し、前後スライドへの影響があれば併せて報告

## 出力

- **形式**: チャット（対話形式で段階的に生成） + 最終ファイル
- **保存先**: `documents/<該当週フォルダ>/` 配下（リポジトリの命名規則に従う）
- **frontmatter**:

```yaml
---
title: "発表タイトル"
status: draft | reviewed | stable
created: YYYY-MM-DD
updated: YYYY-MM-DD
audience: "聴衆の属性"
takeaway:
  - "伝えたいこと1"
  - "伝えたいこと2（複数の場合）"
duration_min: N
venue: "会議室 / 学会 / オンライン"
format: "口頭発表 / ポスター / ライトニングトーク"
source_notes:
  - "ソースノートへの相対パス"
lang: ja | en
---
```

**最終ドキュメント構造**:
```markdown
# [発表タイトル]

## メタ情報
- 聴衆 / 目的 / 時間 / 会場 / 形式

## ストーリー概要
(SCQA + 主張順序の圧縮版)

## スライド別内容
### Slide 1: [ヘッドライン]
...

## 想定Q&A
- Q: ... / A: ...

## 削除候補 / 時間調整ポイント
- Slide X: 時間不足時は省略可能（理由）
```

## このスキル固有の注意点

### 価値の定義（常に問い直す）

1. **聴衆が伝えたいことを理解する**: 「何の話だった？」に答えられる（伝えたいことが少ないほど明確になる）
2. **聴衆の理解/判断/行動が発表前より改善する**: 情報転送ではなく変化を起こす
3. **削除した情報の価値 ≥ 残した情報の価値**: 「詰め込みすぎ」は発表を殺す

**上記3点に貢献しない要素はすべてノイズ。** 形式やスタイルの議論ではなく、この3点への貢献度のみで判断する。

### 表現ルール（全フェーズ共通）

スライド内容・原稿・ツリー・SCQA など、このスキルが生成するすべてのテキストに適用する。

#### 禁止表現

**1. 自己評価的表現**: 成果の価値を自ら主張する表現。結果で示す。

| 禁止 | 代替 |
|:---|:---|
| 核心的貢献、画期的、革新的 | 具体的な技術変更を記述する |
| 優れた、卓越した | 数値比較で示す |
| パラダイムシフト | 「アプローチの変更」等、事実を書く |
| 〜を実現する（未実証のまま） | 「〜を目的とする」または実証結果を書く |

**2. 比喩・装飾表現**: 技術内容を直接記述する。

| 禁止 | 代替 |
|:---|:---|
| 橋渡し、鍵を握る、扉を開く、柱となる | 事実を直接記述する |
| 〜を支える、〜の心臓部 | 「〜に必要な」「〜を構成する」 |
| 〜の世界、〜の風景 | 削除 |
| ナビゲートする、駆動する | 「探索する」「計算する」等の動作動詞 |

**3. 曖昧な量・程度の表現**: 数値・具体的比較に置き換える。

| 禁止 | 代替 |
|:---|:---|
| 非常に、極めて、かなり、大幅に | 具体的な数値比較 |
| 飛躍的に、劇的に | 変化量を数値で示す |
| 多くの、少ない、一部の | 具体的な個数・割合 |
| 高い、低い、大きい、小さい | 数値+単位+基準 |
| 約、およそ（範囲なし） | 具体値または明示的な範囲 |

**4. 演出的な接続・前置き**: 削除し、事実をそのまま書く。

| 禁止 | 代替 |
|:---|:---|
| ここで重要なのは、注目すべきは | 削除。事実をそのまま書く |
| 興味深いことに、驚くべきことに | 削除。結果を提示する |
| 言うまでもなく、当然のことながら | 削除。根拠が自明なら書かなくてよい |
| まず理解すべきは | 削除。説明を始める |
| 本スライドでは〜を説明する | 削除。スライドタイトルが内容を示す |

**5. 根拠のない評価・常套句**: 評価は定量的根拠とセットでのみ使用可。

| 禁止 | 代替 |
|:---|:---|
| 効果的、有用、有望 | 何がどう変わるかを具体的に書く |
| 一般的に〜と言われている（出典なし） | 出典を付けるか削除 |
| 近年〜が注目されている（出典なし） | 具体的な時期と出典を示す |
| 広く知られているように | 削除。事実を書く |

**6. 繰り返し・冗長表現**: 同じ情報を別の言い方で繰り返さない。

| 禁止 | 代替 |
|:---|:---|
| 同じ主張を言い換えて複数回書く | 1回で明確に書く |
| 「前述のとおり〜」で再説明する | セクション番号を参照する |
| 「新しく新規な」等の同義反復 | 片方を削除 |

**7. 造語・未定義用語**: 既存の用語を使うか、初出時に定義する。

| 禁止 | 代替 |
|:---|:---|
| 引用根拠のない専門用語風の造語 | 既存の用語を使う |
| 省略形（初出で未定義） | 初出時に正式名称と定義を提示 |

#### 必須ルール

1. **事実と推論を明示的に区別する**
   - 事実: 「仕様 TS 38.211 §4.3.2 では〜と定義されている」
   - 推論: 「この制約から〜と考えられる」
   - 未検証: 「〜の可能性があるが、確認が必要」
2. **数値は可能なら比較対象・条件とセットで提示する**（努力目標。冗長になる場合は省略してよい）
   - 「処理量が増加する」より「処理量が O(MK) から O(MK²) に増加する」の方が望ましい
   - **ただし、出典や根拠のない数値を捏造してはならない**。具体的な数値が不明なら「増加する」のまま残す方がよい
3. **用語は既存の定義を使う**。造語が必要な場合は初出時に定義する
4. **演出的な言い回しはユーザーが指定しない限り使わない**
5. **各文は新しい情報を追加する**。情報量ゼロの文を書かない

### フレームワーク（運用上の参照）

| フレームワーク | 要点 |
|:---|:---|
| Pyramid Principle | 結論→主張→根拠。トップダウン。兄弟は MECE |
| Assertion-Evidence | 1スライド=1主張（完全文ヘッドライン）+視覚的根拠 |
| Doumont 3原則 | 聴衆適合 / S/N比最大化 / 効果的冗長性。並列3個以内 |
| Peyton Jones | 伝えたいことを絞る、例必須、pruning 徹底 |
| SCQA | Situation → Complication → Question → Answer |

原典を読むとさらに理解が深まる（特に Alley と Doumont は工学発表者にとって実用的）:
- Barbara Minto, *The Pyramid Principle* (Pearson)
- Michael Alley, *The Craft of Scientific Presentations* (Springer)
- Jean-luc Doumont, *Trees, Maps, and Theorems* (Principiae)
- Simon Peyton Jones et al., "How to give a good research talk", *SIGPLAN Notices* 28(11), 1993

### 外部リサーチのルール

以下の場合のみ外部検索を実行する:
- 提供資料にない最新情報が主張に必要
- 競合手法の性能数値が比較に必要
- 業界動向・規格状況の確認が必要

制約: 最小限に留め、出典明示、推測と検索結果を混同しない。
時間が限られている場合は、検索前にユーザーに確認する。

### 3GPP研究コンテキストとの接続

- ソースとして `documents/` のノートを直接参照可能
- 技術的な根拠には原則1（出典記録）を適用する
- 実装制約ギャップ（原則2）は発表の差別化ポイントになりうる
- ノートの `axes:` 情報を活用して聴衆への文脈付けを行う

## 更新のしかた

発表を1回終えるごとに、以下をメモしてスキルを改善する:

1. **うまくいったこと**: どのルールが効いたか
2. **うまくいかなかったこと**: どのルールが現場に合わなかったか
3. **追加したいルール**: 今回の経験から導出された新ルール
4. **削除/修正したいルール**: 不要と判明したもの
